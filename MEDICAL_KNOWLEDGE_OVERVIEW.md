# Medical Knowledge Demo Service Overview

## 전체 개요

이 프로젝트는 **학습용 의료 QA 데이터**를 기반으로 Milvus에 지식 DB를 구축하고, 그 위에 **REST API**와 **웹 서비스**를 연결한 데모 시스템입니다.

단순히 데이터를 저장하는 것이 목적이 아니라, 아래와 같은 실제 활용 시나리오를 염두에 두고 구성했습니다.

- 사용자가 자연어 질문으로 의료 지식을 빠르게 찾을 수 있어야 함
- 의미 기반 검색과 키워드 기반 검색을 함께 지원해야 함
- 진료과, 언어, 문제 유형 같은 조건으로 결과를 좁힐 수 있어야 함
- 발표나 시연 시 복잡한 셸 명령 없이 PyCharm에서 바로 실행 가능해야 함
- Milvus 서버 주소만 바꿔도 다른 환경으로 쉽게 옮길 수 있어야 함

즉, 이 프로젝트는 **의료 QA 데이터를 검색 가능한 지식 자산으로 바꾸고**, 이를 **API와 웹 UI까지 연결한 전체 흐름**을 보여주기 위한 구조입니다.

---

## 왜 이런 데이터를 사용했는가

이번 구성의 출발점은 `medical_knowledge/essential` 안의 **학습용 의료 QA 데이터**입니다.

이 데이터를 사용한 이유는 다음과 같습니다.

### 1. 정답이 있는 구조화된 의료 데이터이기 때문

학습용 데이터는 일반 문서와 달리 아래처럼 구조가 분명합니다.

- 질문 `question`
- 정답 `answer`
- 식별자 `qa_id`
- 분류 정보 `domain`, `q_type`
- 상위 카테고리 역할을 하는 진료과 폴더

이런 구조는 지식 DB 구축에 매우 유리합니다.  
검색 결과를 단순 텍스트 조각이 아니라 **질문-정답 쌍**으로 보여줄 수 있고, 이후 RAG나 QA 서비스로 확장하기도 쉽습니다.

### 2. 의료 도메인 특성상 정확한 근거 단위가 중요하기 때문

일반 문서 검색에서는 긴 문서를 잘라 저장해도 되지만, 의료 데이터는 문맥과 정답의 연결이 중요합니다.  
학습용 QA 데이터는 이미 **질문에 대한 정답 단위**로 정리되어 있으므로, 검색 결과를 사용자에게 설명하기 좋고 시연에도 적합합니다.

### 3. 카테고리 정보가 이미 존재하기 때문

데이터는 진료과별로 분리되어 있습니다.

- 내과
- 산부인과
- 소아청소년과
- 응급의학과

이 구조는 Milvus에서 **파티션 전략**을 적용하기에 적합합니다.  
즉, 이미 데이터셋 자체가 “검색 범위를 나눌 수 있는 기준”을 가지고 있었기 때문에 파티션 설계가 자연스럽게 이어졌습니다.

### 4. 실제 검색 서비스 형태로 확장하기 좋은 데이터이기 때문

이 데이터는 단순 분석용이 아니라, 아래 같은 서비스 확장성이 있습니다.

- 질의응답 검색
- 의료 교육용 검색 시스템
- 하이브리드 검색 데모
- RAG 기반 응답 생성의 지식 소스

따라서 이번 프로젝트는 단순 적재가 아니라, **실서비스로 이어질 수 있는 형태**를 목표로 설계했습니다.

## 왜 Milvus 기반 지식 DB로 구성했는가

의료 QA 데이터를 파일로만 보관하면 사람이 직접 찾아야 합니다.  
검색 가능한 시스템으로 바꾸려면 아래 기능이 필요합니다.

- 의미가 비슷한 질문 찾기
- 단어가 정확히 일치하는 문장 찾기
- 메타데이터로 필터링
- 빠른 응답 속도

Milvus를 사용한 이유는 다음과 같습니다.

### 1. 벡터 검색에 적합하기 때문

사용자는 항상 데이터에 있는 질문 문장을 그대로 입력하지 않습니다.  
예를 들어 표현이 달라도 의미가 비슷하면 찾아줘야 합니다.

이 요구를 만족하려면 **dense vector 기반 의미 검색**이 필요합니다.

### 2. 의료 검색은 키워드도 중요하기 때문

의료 질문은 특정 검사명, 질환명, 약물명처럼 **정확한 용어 일치**가 중요합니다.  
의미 검색만으로는 이런 부분이 약해질 수 있기 때문에 **sparse vector 기반 검색**도 함께 필요합니다.

그래서 이번 시스템은 다음 두 방식을 같이 사용합니다.

- Dense 검색: 의미 유사도 중심
- Sparse 검색: 단어/용어 일치 중심

즉, Milvus는 단순 벡터 DB가 아니라 **하이브리드 검색 엔진의 핵심 저장소** 역할을 합니다.

---

## 왜 Dense + Sparse 하이브리드 검색으로 구성했는가

이 부분은 이번 설계에서 가장 중요한 이유 중 하나입니다.

### Dense 검색만 사용하면 부족한 점

Dense 검색은 문장의 뜻이 비슷한 경우 강합니다.

- 표현이 달라도 비슷한 질문을 찾기 좋음
- 자연어 검색 경험이 좋음

하지만 의료 분야에서는 아래 문제가 있습니다.

- 특정 용어가 정확히 일치해야 하는 경우가 많음
- 약어, 검사명, 질환명, 수치 기준이 중요함
- 드문 전문용어는 의미 임베딩만으로 놓칠 수 있음

### Sparse 검색만 사용하면 부족한 점

Sparse 검색은 단어 일치에는 강합니다.

- 키워드 매칭이 명확함
- 전문용어 검색에 유리함

하지만 아래 한계가 있습니다.

- 표현이 조금만 바뀌어도 놓칠 수 있음
- 자연어 질문에 대한 유연성이 떨어짐

### 그래서 두 방식을 함께 사용함

의료 QA 검색은 아래 두 요구를 동시에 만족해야 합니다.

- 질문의 뜻이 비슷하면 찾을 것
- 핵심 용어가 정확히 들어가면 더 잘 찾을 것

그래서 하이브리드 검색이 가장 자연스럽습니다.

- Dense: 문맥적 유사성 확보
- Sparse: 핵심 키워드 보강
- WeightedRanker: 두 결과를 함께 반영

즉, 이번 시스템은 “의미 검색”과 “용어 검색”을 같이 만족시키기 위해 하이브리드 구조로 설계했습니다.

## 왜 메타데이터를 따로 저장했는가

질문과 정답만 저장해도 검색은 가능하지만, 실제 서비스나 시연에서는 그것만으로 부족합니다.

메타데이터를 추가한 이유는 다음과 같습니다.

### 1. 검색 결과를 좁히기 위해서

사용자는 종종 전체 데이터가 아니라 특정 조건만 보고 싶어합니다.

예:

- 내과 문제만 보고 싶다
- 소아청소년과만 찾고 싶다
- 특정 domain만 찾고 싶다
- 특정 문제 유형만 보고 싶다

이런 요구를 위해 아래 필드를 저장했습니다.

- `department`
- `language`
- `domain`
- `q_type`

### 2. 결과 설명력을 높이기 위해서

발표나 시연에서는 검색 결과가 “왜 이 결과인지” 보여주는 게 중요합니다.

예:

- 어느 진료과 데이터인지
- 어떤 유형의 질문인지
- 어떤 파일에서 왔는지

그래서 아래 같은 표시가 가능해집니다.

- 결과 소속 진료과
- 언어 정보
- 원본 파일 경로

### 3. API와 웹 필터 기능을 쉽게 만들기 위해서

메타데이터가 저장되어 있으면 API에서 expr 필터를 만들 수 있고, 웹 화면에서는 드롭다운 필터를 쉽게 구성할 수 있습니다.

즉, 메타데이터는 단순 부가 정보가 아니라 **검색 제어와 설명 가능성**을 위해 필요합니다.

---

## 왜 파티션을 카테고리 기준으로 나눴는가

Milvus는 하나의 컬렉션 안에서도 파티션을 나눌 수 있습니다.  
이번 프로젝트에서 진료과를 기준으로 파티션을 나눈 이유는 다음과 같습니다.

### 1. 데이터 자체가 이미 진료과 구조를 가지고 있기 때문

원본 데이터가 진료과 폴더별로 정리되어 있으므로, 이를 그대로 검색 구조에 반영하는 것이 가장 자연스럽습니다.

### 2. 검색 범위를 줄이기 위해서

예를 들어 사용자가 산부인과 질문만 찾고 싶다면 전체 데이터를 모두 뒤질 필요가 없습니다.  
산부인과 파티션만 대상으로 검색하면 더 명확하고 효율적입니다.

### 3. 발표 시 구조 설명이 쉬워지기 때문

“하나의 컬렉션 안에서 진료과별 파티션으로 나눴다”는 설명은 구조적으로 이해하기 쉽고, 데이터가 어떻게 조직되었는지를 보여주기 좋습니다.

즉, 파티션은 단순 분리가 아니라 **카테고리 기반 검색 최적화와 설명 용이성**을 위해 사용했습니다.

## 왜 BM25 통계 파일이 필요한가

프로젝트 안에는 `medical_knowledge_hybrid_bm25_stats.json` 파일이 생성됩니다.

이 파일이 필요한 이유는 sparse 검색이 단순 단어 목록이 아니라 **BM25 스타일 가중치**를 사용하기 때문입니다.

파일에는 아래 정보가 저장됩니다.

- 전체 문서 수
- 평균 문서 길이
- 각 토큰이 몇 문서에 등장했는지
- BM25 파라미터

이 통계는 아래 두 곳에서 동일하게 사용됩니다.

- 문서 저장 시 sparse vector 생성
- 검색 시 query sparse vector 생성

즉, 이 파일은 sparse 검색의 기준표 역할을 하며,  
같은 기준으로 문서와 query를 벡터화하기 위해 필요합니다.

## 1. 데이터 전처리 및 스키마 구성 및 저장

데이터 설정은 `app/config.py` 에 모여 있습니다. 발표나 배포 시에는 `.env`에서 `MILVUS_HOST`, `MILVUS_PORT`, `DATA_ROOT` 정도만 바꾸면 됩니다.

기본적으로 학습 데이터는 아래 경로를 읽습니다.

- `medical_knowledge/essential/Training/02.라벨링데이터`

전처리는 `app/data_loader.py` 에서 담당합니다. 여기서 하는 일은 아래와 같습니다.

- JSON 파일 순회
- `question`, `answer`, `qa_id`, `domain`, `q_type` 추출
- `department`를 폴더명 기준으로 추출
- `language`를 `ko`, `en`, `mixed`, `unknown`으로 판별
- `partition_name`을 Milvus에서 안전하게 쓸 수 있는 ASCII 이름으로 생성
- 검색용 본문 `content`를 `Question + Answer` 형태로 구성
- 메타데이터를 `metadata` JSON으로 정리

저장용 스키마는 `app/search_service.py` 에 정의되어 있습니다. 하나의 컬렉션 안에 다음 필드를 둡니다.

- `doc_id`
- `qa_id`
- `department`
- `partition_name`
- `language`
- `domain`
- `q_type`
- `question`
- `answer`
- `content`
- `source_path`
- `metadata`
- `dense_vector`
- `sparse_vector`

DB 구축 실행은 `build_db.py` 가 담당합니다. 실행 시 내부적으로 아래 순서로 진행됩니다.

1. 학습용 데이터 로드
2. BM25 통계 생성
3. Milvus 컬렉션 생성
4. 파티션 생성
5. dense/sparse 벡터 생성
6. 배치 단위 insert
7. flush/load
8. BM25 통계 파일 저장

## 2. API 상세 내용

REST API는 `app/api_app.py` 에 구현되어 있습니다. 실행은 `run_api.py` 로 합니다.

기본 실행 후 Swagger 문서는 아래 주소에서 확인할 수 있습니다.

- `http://127.0.0.1:8001/docs`

제공하는 엔드포인트는 다음과 같습니다.

- `GET /health`
- `GET /filters`
- `POST /search`

### 왜 API 레이어를 따로 두었는가

데모라 하더라도 DB에 직접 붙는 구조만으로는 재사용성과 확장성이 낮습니다.  
API를 두면 아래 이점이 있습니다.

- 웹, 앱, 외부 시스템이 같은 검색 기능을 재사용 가능
- 검색 조건을 표준화할 수 있음
- 프론트엔드와 검색 로직을 분리 가능
- 추후 인증, 로깅, 캐싱, 모니터링 추가가 쉬움

### API 동작 방식

`POST /search`는 아래 요청 값을 받습니다.

- `query`
- `limit`
- `department`
- `language`
- `domain`
- `q_type`
- `dense_weight`
- `sparse_weight`

검색 로직은 `app/search_service.py` 의 `search_documents()` 에 있습니다. 동작 방식은 아래와 같습니다.

1. Milvus 연결
2. BM25 통계 로드
3. query를 dense 벡터로 변환
4. query를 sparse 벡터로 변환
5. 메타데이터 필터 expr 생성
6. 필요하면 특정 department 파티션만 대상으로 검색
7. `Collection.hybrid_search()` 수행
8. 결과를 JSON 형태로 반환

반환 결과에는 아래 정보가 포함됩니다.

- `score`
- `doc_id`
- `qa_id`
- `department`
- `language`
- `domain`
- `q_type`
- `question`
- `answer`
- `source_path`

즉 API는 검색 엔진 역할을 수행하며, 클라이언트는 복잡한 Milvus 세부 구현을 몰라도 됩니다.

## 3. Web 서비스 상세 내용

웹 데모는 `app/web_app.py` 와 `app/templates/index.html` 로 구성되어 있습니다. 실행은 `run_web.py` 로 합니다.

기본 주소는 아래입니다.

- `http://127.0.0.1:8002`

웹 서비스의 목적은 복잡한 API 호출 없이도 하이브리드 검색 흐름을 시연할 수 있게 하는 것입니다.

화면은 다음 요소로 이루어져 있습니다.

- 검색어 입력 영역
- 검색 개수 입력
- 진료과 필터
- 언어 필터
- domain 필터
- q_type 필터
- 결과 카드 리스트

### 왜 웹 화면을 따로 만들었는가

REST API만 있으면 개발자에게는 충분하지만, 발표나 시연에서는 비개발자도 바로 이해할 수 있는 화면이 필요합니다.

웹 UI를 둔 이유는 다음과 같습니다.

- 검색 과정을 눈으로 보여줄 수 있음
- 필터가 어떻게 동작하는지 직관적으로 설명 가능
- 결과 품질을 질문/정답 카드로 바로 시연 가능
- “DB 구축 → API → 사용자 화면” 전체 흐름을 한 번에 보여줄 수 있음

### 웹 동작 흐름

1. 첫 화면 접속 시 필터 옵션을 불러옵니다.
2. 사용자가 질문과 필터를 입력합니다.
3. 서버가 내부적으로 `search_documents()`를 호출합니다.
4. 결과를 HTML 카드 형태로 렌더링합니다.
5. 각 카드에 질문, 정답, 메타데이터, 소스 경로를 보여줍니다.

발표용 관점에서 웹 화면의 의미는 다음처럼 설명하면 좋습니다.

- 사용자는 단순한 검색 UI만 보지만
- 내부적으로는 dense + sparse 하이브리드 검색이 수행되고
- 메타데이터 필터와 파티션 제한이 함께 적용되며
- 결과는 의료 QA 지식 DB에서 바로 반환됩니다.
